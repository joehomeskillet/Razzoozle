# i18n-guard — SDD (deterministisches i18n-CI-Gate + Subagent)

**Date:** 2026-07-15
**Status:** Design Phase — flood-ready, WP-Specs liegen in `scratchpad/next-session-wp-specs/IG-*.md`
**Grundlage:** GitHub-Research-Empfehlung `lingualdev/i18n-check` (npm `@lingual/i18n-check`, MIT, standalone CLI, kein API-Key). Ergänzt den bestehenden LLM-Werte-Übersetzer `scripts/locale-sync.mjs` — ersetzt ihn NICHT.
**Stakeholder:** joel.scherrer90@gmail.com

## Ziel + Nicht-Ziele

**Ziel:** Ein deterministisches CI-Gate um den bestehenden Locale-Stack:
`@lingual/i18n-check@0.9.5` findet **missing keys** (gegen Source `de`), **invalid keys**
(Interpolations-/Typ-Drift) blockierend, plus **unused/undefined keys** als
Advisory-Report. Dazu: CODEOWNERS-Schutz auf `packages/web/src/locales/**` und eine
Agent-Definition `i18n-guard` (Report-only, delegiert Fixes an locale-sync).

**Nicht-Ziele:**
- KEIN Ersatz von `scripts/locale-sync.mjs` (bleibt der einzige Schreibweg für Locale-JSON)
  und KEIN Umbau von `scripts/check-locales.sh` / `rust/gate.sh` (Koexistenz, s.u.).
- KEINE Matrix locale×namespace im Workflow: der komplette Check läuft in **0,05 s**
  (empirisch verifiziert) — eine Matrix wäre reiner Overhead (YAGNI).
- `unused`/`undefined` werden NIE blockierend (empirisch ~440 unused- und ~80
  undefined-Findings, großteils False Positives, s. Empirie).
- KEIN Key-Löschen aus dem unused-Report ohne Menschen-Review (Memory: locale-sync
  "Never delete keys").
- KEIN separates `.github/workflows/i18n-check.yml` — GitHub ist Push-Mirror ohne
  aktive CI; das Gitea-Workflow-File ist mirror-safe geschrieben (SHA-pinned
  `uses:`), sodass ein späteres Kopieren nach `.github/workflows/` 1:1 funktioniert.

## Empirie (i18n-check@0.9.5 gegen den echten Baum, 2026-07-15)

Gemessen mit `npx -y @lingual/i18n-check@0.9.5 -l packages/web/src/locales -s de -f i18next`:

1. **missing+invalid (Default-Checks): Exit 1, 10 echte fehlende Keys** —
   `game.playerNotFound` (es/fr/it/zh errors.json), `game.locked` (zh errors.json),
   `profile.intro` (en/es/fr/it/zh manager.json). Invalid: 0. Laufzeit 0,05 s.
   → Das blockierende Gate wäre HEUTE rot: **IG-1 (Backfill) muss vor/mit IG-3
   (Workflow) auf main sein**, sonst ist der neue Check ab Tag 1 rot.
2. **undefined (`-o undefined -u packages/web/src`): Exit 1, ~80 Findings** —
   enthält False Positives, weil i18n-check die Namespace-Bindung
   (`useTranslation("errors")` → `t("auth.invalidInput")`) nicht auflöst
   (verifiziert: `auth.*` existiert in `de/errors.json`, wird aber als undefined
   gemeldet). → nur Advisory.
3. **unused (`-o unused -u packages/web/src`): Exit 1, ~440 Findings** —
   enthält False Positives durch dynamische Keys (verifiziert: `language.de` …
   `language.zh` werden via Template-String in `LanguageSwitcher.tsx` benutzt,
   aber als unused gemeldet). → nur Advisory, Löschungen nur mit Menschen-Review.

**Konsequenz (Gate-Design):** Blocking = NUR `missingKeys` + `invalidKeys`
(deterministisch, 0 False Positives beobachtet). `unused` + `undefined` =
Advisory-Step mit `continue-on-error: true` und Input für den i18n-guard-Agenten.

## Architektur-Entscheidungen

### E1 — devDependency am ROOT, nicht in packages/web
`check-locales.sh` und `locale-sync.mjs` leben in root `scripts/`; i18n-check prüft
den Locale-Baum UND scannt `packages/web/src` — ein repo-weites Tooling wie
oxlint/playwright, die ebenfalls root-devDeps sind. Exakt gepinnt: `"@lingual/i18n-check": "0.9.5"`
(kein Caret — CI-Gate-Tool, Flag-/Verhaltens-Drift unerwünscht). Bin-Name: `i18n-check`.

### E2 — Eingefrorener Script-Contract (Wave-0, gilt für ALLE WPs)
Root `package.json` scripts (Owner: IG-2; IG-3 + IG-5 referenzieren sie nur):

```json
"i18n:check":  "i18n-check -l packages/web/src/locales -s de -f i18next",
"i18n:report": "i18n-check -l packages/web/src/locales -s de -f i18next -o unused,undefined -u packages/web/src"
```

Semantik: `i18n:check` = blockierend (Exit 1 bei missing/invalid).
`i18n:report` = Exit 1 bei Findings, wird deshalb IMMER advisory konsumiert
(`continue-on-error: true` in CI, `|| true` im Agenten).

### E3 — check-locales.sh: KOEXISTENZ, unverändert (0 LOC Änderung)
Begründung, warum NICHT auf i18n-check umgestellt wird:
1. `rust/gate.sh` ruft `check-locales.sh` in bare Worktrees **ohne node_modules**
   auf — genau dafür ist es dependency-frei (python3 + node stdlib) gebaut.
   i18n-check braucht installierte devDeps; ein "wenn vorhanden, dann"-Conditional
   machte das Gate umgebungsabhängig = nicht-deterministisch (verletzt das
   gate.sh-Designprinzip "source of truth, never trust environment").
2. Komplementäre Checks: `locale-sync.mjs check` prüft Deep-Parität gegen die
   **Union aller Locales** (fängt Extra-Keys, die nur in en existieren);
   i18n-check prüft gegen **Source de** + Code-Nutzung. Beides zusammen > eines.
3. Der CI-Workflow (IG-3) ruft BEIDE: erst `bash scripts/check-locales.sh`
   (Validität+Parität), dann `pnpm run i18n:check` (missing/invalid vs. de).

### E4 — CODEOWNERS im Repo-ROOT (ein File für beide Plattformen)
Gitea liest CODEOWNERS aus root/`.gitea/`/`docs/`, GitHub aus root/`.github/`/`docs/`
→ root ist der einzige Ort, den beide lesen. Owner-Einträge: `@agent-claude`
(Gitea-Repo-Owner, dort gültig) + `@joehomeskillet` (GitHub-Mirror-Owner, dort
gültig); jede Plattform ignoriert ihr unbekannte Namen. Effekt auf Gitea:
Auto-Review-Request auf PRs, die `packages/web/src/locales/**` anfassen.

### E5 — Agent i18n-guard = Report-only-Wächter
`docs/agents/i18n-guard.md` nach dem Muster von `docs/agents/locale-sync.md`.
Modus `check`: `pnpm run i18n:check` + `pnpm run i18n:report || true` fahren,
Findings klassifizieren (echte Lücke vs. Dynamik-/Namespace-False-Positive),
Report ausgeben. Modus `fix`: NIE selbst JSON editieren — missing keys als
locale-sync-Auftrag formulieren (diff→translate→apply-Workflow), unused-Kandidaten
nur als Menschen-Review-Liste. Arbeitsteilung: **i18n-check findet, locale-sync
schreibt, der Mensch löscht.**

### E6 — Workflow-Design (IG-3)
Eigenes `.gitea/workflows/i18n-check.yml` (NICHT in ci.yml einbauen — ci.yml ist
4-stufig mit needs-Kette; ein orthogonales 30-s-Gate soll PRs nicht an die
unit-Stage ketten und unabhängig re-runnable sein). Pattern 1:1 aus ci.yml
übernommen: SHA-pinned checkout v5.0.1, corepack, `CI=true pnpm install
--frozen-lockfile --ignore-scripts`, concurrency cancel-in-progress,
`permissions: contents: read`. Trigger: push main + pull_request mit
paths-Filter (`packages/web/src/**`, `package.json`, `pnpm-lock.yaml`,
`scripts/check-locales.sh`, `scripts/locale-sync.mjs`, das Workflow-File selbst)
+ `workflow_dispatch` für manuellen Smoke.

## WP-Schnitt (file-disjunkt, alle parallel-fähig)

**Split-Check:** Umfang → **5 WPs** (IG-1 Locale-Backfill · IG-2 devDep+Scripts ·
IG-3 Workflow · IG-4 CODEOWNERS · IG-5 Agent-Def). Jedes WP = 1 Datei-Cluster
< 150 LOC Diff, file-disjunkt, Contract (E2) hier eingefroren → alle 5 in EINER
Welle parallel (≥3-Worker-Ziel erfüllt). Tests: kein eigenes Test-WP nötig —
IG-1/IG-2 werden durch die bestehenden deterministischen Gates
(check-locales.sh, i18n:check-Exit-Code) abgenommen, IG-3/IG-4/IG-5 sind
deklarative Einzeldateien ohne Test-Oberfläche (YAGNI).

| WP | Datei(en) | Lane-Empfehlung | Effort |
|---|---|---|---|
| IG-1 Locale-Backfill (10 Keys) | `packages/web/src/locales/{es,fr,it,zh}/errors.json`, `{en,es,fr,it,zh}/manager.json` — NUR via locale-sync.mjs | locale-sync (haiku, in-harness, isolation:worktree) | S |
| IG-2 devDep + i18n-Scripts | `package.json` (root), `pnpm-lock.yaml` | codex-gpt5 | S |
| IG-3 Gitea-Workflow | `.gitea/workflows/i18n-check.yml` (neu) | grok-build (+ SECURITY-CHECK-Schlusszeile) | S |
| IG-4 CODEOWNERS | `CODEOWNERS` (root, neu) | Free-Pool (zen-coder / ali-coder) | XS |
| IG-5 Agent-Definition | `docs/agents/i18n-guard.md` (neu) | Free-Pool (or-text-free) / docs-writer | S |

Details je WP: `scratchpad/next-session-wp-specs/IG-{1..5}-*.md` (agent-fertig,
cold-startbar, inkl. Worktree-Setup, Gates, Commit-Message).

## Dispatch-Plan

- **Wave-0 (bereits erledigt, dieses Dokument):** Contract-Freeze E2 (Script-Namen,
  Version 0.9.5, Datei-Orte). Kein eigener Worker nötig.
- **Wave-1 parallel (5 Worker):** IG-1…IG-5, jeder im eigenen Worktree.
  Subscription-first: codex (IG-2), grok (IG-3), locale-sync-Harness-Agent (IG-1),
  Free-Pool (IG-4, IG-5). **≥2 Free-Pool-Writer in der Welle → grok-Wave-Review
  des kombinierten Diffs PFLICHT vor Merge** (`WAVE-REVIEW: CLEAN|FINDINGS(n)`).
- **Review:** Reviewer ≠ Fix-Agent, cross-vendor, adversarisch. IG-3 (CI-Datei =
  supply-chain-sensibel: SHA-Pins, permissions, keine Secrets) zusätzlich von
  codex gegengelesen.
- **Merge-Reihenfolge (Fable merged, git -C main-tree):**
  IG-1 → IG-2 → IG-3 → IG-4/IG-5 (beliebig). IG-1+IG-2 MÜSSEN vor IG-3 auf main
  sein, sonst ist der erste i18n-check-Run rot (10 bekannte missing keys) bzw.
  `pnpm run i18n:check` unbekannt.
- **Abnahme nach Merge:** (a) `workflow_dispatch` auf i18n-check.yml → grün;
  (b) Kontroll-PR mit absichtlich fehlendem de-Key-Backfill → Workflow ROT
  (Negativ-Probe); (c) PR gegen `packages/web/src/locales/**` → CODEOWNERS
  requested Review; (d) `pnpm run i18n:check` lokal Exit 0.
- **Nach Abnahme:** `routing-outcome record` pro Worker; Memory-Update
  (i18n-Gate live, Agent-Def-Fundort).

## Risiken / offene Punkte

- **i18n-check-Flag-Drift:** 0.9.5 exakt gepinnt; Flags empirisch verifiziert
  (`--help`-Dump in den WP-Specs). Upgrade nur bewusst mit Re-Verifikation.
- **undefined-False-Positives:** Namespace-Bindung wird nicht aufgelöst → bleibt
  advisory. Falls lingualdev das später kann: Advisory→Blocking als eigenes
  Follow-up-WP (nicht hier).
- **Gitea paths-Filter:** funktioniert ab Gitea ≥1.20; falls die Instanz das
  Event trotzdem breit feuert, ist der Job mit ~60 s (install-dominiert) billig.
- **CODEOWNERS-Enforcement:** Gitea requested Reviews, erzwingt sie aber nur mit
  Branch-Protection ("require approvals") — Branch-Protection-Änderung ist
  bewusst NICHT Teil dieser Welle (Nicht-Ziel, Admin-Entscheid des Users).

## Events

Namenskonvention pro Map: `class:*` (camelCase, Rust `pub mod class` in `rust/protocol/src/constants.rs`), `results:*` analog, User-CRUD bisher REST-only (`/api/users/:id/<verb>`). Neue Bulk-Events zuerst in `rust/protocol/src/constants.rs` + `packages/common/src/constants.ts` einfrieren (Contract-Freeze), dann Handler/DB-Fn.

- `results:bulkDelete` — Payload `Vec<String>` (ids), owner-scoped, analog `results:delete`.
- `class:bulkDelete` — Payload `Vec<i64>`; muss Memberships entfernen ohne den Orphan-Delete-Trigger auszulösen (siehe classes-Zeile).
- `class:setActive` / `class:bulkSetActive` — `{id(s), active}`.
- `class:setStudentActive` / `class:bulkSetStudentActive` — analog `class:regenPin`-Vorbild, `{studentId(s), active}`.
- `class:bulkDeleteStudent` — analog `class:removeStudent`, gebündelt.
- `class:bulkAssignStudent` — analog `class:moveStudent`, gebündelt. ⚠ VERIFY Namensschema gegen finale SDD abstimmen.
- Users-Bulk: kein Socket-Namespace für Admin-User-CRUD vorhanden — REST bleibt Muster: `POST /api/users/bulk-activate|bulk-deactivate|bulk-delete` (kebab-case wie bestehend `/api/users/:id/disable|enable|reset-password`). ⚠ VERIFY: Backend-Map lässt offen, ob Socket- oder HTTP-Bulk-Namensraum gewünscht ist.
- Gamemode/Play: keine neuen Events nötig — bestehendes `manager:setGameConfig`/`game:create` deckt beide WPs ab.

## 3. Player JOIN entry — clarification

**The task premise needs correcting: `/submit` is NOT the player join entry.**

- The actual join entry is **`/` (`pages/(auth)/index.tsx`)**, which renders `Room` (PIN input, `features/game/components/join/Room.tsx`) and — once `usePlayerStore().player` exists — `Username` (`features/game/components/join/Username.tsx`). Confirmed at `pages/(auth)/index.tsx:28-30` (`if (player) return <Username />`).
- `/submit` (`pages/submit/index.tsx` → `features/submission/SubmitPage/SubmitPage.tsx`) is a **standalone crowd-sourced quiz-question submission form** — a visitor fills in a question they'd like added to the pool (`submittedBy` name field is the *question author's* name, not a player identity), and it POSTs via `socket.emit(MANAGER.SUBMIT_QUESTION, …)` for host moderation (`SubmitPage.tsx:1,97-147`). It has no PIN field, no `PLAYER.JOIN`/`PLAYER.LOGIN` emit, and no relationship to a live game session. `Room.tsx` links to it as a footer "public entry point to the question-submission page" (`Room.tsx:117-124`), which is the only connection between the two flows — a cross-link, not a shared flow.

How `/` actually takes name/code today:
1. **Code:** `PinInput` (6-char, `features/game/components/join/Room.tsx:18,101-105`) or a `?pin=` deep-link search param (`Room.tsx:27,85-92`) → `socket.emit(PLAYER.JOIN, code)`.
2. **Name:** free-text `Input` (max 20 chars, `Username.tsx:17,91-111`) → `socket.emit(PLAYER.LOGIN, { gameId, data: { username, avatar, identifier? } })`. The `identifier` field is a **free-text optional** box that only appears when the server's `SUCCESS_ROOM` payload carries `requireIdentifier: true` (`Username.tsx:68-74,120-137`) — it is not validated, not a class-roster picker, and not an emoji-PIN.

## 4. Gaps found (grounding for `05-class-mode-join-spec.md`)

- **Host-side per-game "Klassen-Modus" switch already exists**: `ConfigSelectQuizz.tsx:29,90-93,254-265` — a `ToggleField` shown when the manager account has `config.klassenEnabled === true`, wired into `socket.emit(GAME.CREATE, { quizzId, selectedModes: { klassen: klassenMode, … } })`. This satisfies the charter's "Host-Switch vor Spielstart" requirement at the UI level. `klassenEnabled` itself is a separate, account-wide manager setting (`features/manager/components/configurations/ConfigGameMode.tsx:62-96,176-179,419`) that also gates catalog labels, question-type visibility (Mathematik/Wortarten/Vokabelliste), and the student/class admin (`schueler`) tab — it is not per-game.
- **Player-side class-roster join UI does not exist**: the only class-mode-adjacent surface in the join flow is the generic optional `identifier` free-text field (`Username.tsx:26,120-137`), gated on `requireIdentifier`, not on `klassenMode`/class membership. There is no name-picker sourced from the class roster and no emoji-PIN entry anywhere in `pages/(auth)/*` or `features/game/components/join/*`.
- **Emoji-PIN infrastructure exists but is not wired into player join**: `PinDialog.tsx` (`features/manager/components/configurations/schueler/PinDialog.tsx`) uses the same `PinInput` primitive to *assign* PINs to students in the admin/teacher console — it is not reachable from the player-facing join path.
- **Open question for `05`/`06`:** whether the server maps `selectedModes.klassen` → `requireIdentifier` (or a distinct server-side flag) at game-creation time is a Rust-server concern outside `packages/web`; this doc only establishes that the **client** has no code path that would let a class-roster pick + PIN verification happen even if the server sent the right signal today. Needs confirming against `rust/server/src/http/emoji_pin.rs` / `socket/manager/classes.rs` (charter §Reuse-Scan) in `05-class-mode-join-spec.md`.
- **Two silent-degrade error paths in Solo** (not present in Multiplayer, which always toasts): a failed `check-answer` POST is treated as a wrong answer with no user-visible error (`features/game/stores/solo.ts:259-276`), and a failed `solo-score` POST is swallowed entirely (`features/game/stores/solo.ts:329-331`). Worth a UX-audit callout for `03-solo-ux-audit.md`.
- **Unknown `GAME.STATUS` name is a silent no-op on `/party/$gameId`** (player route has no `console.warn` fallback, unlike `useManagerGameSession`) — `pages/party/$gameId.tsx:175-178` vs. `features/game/hooks/useManagerGameSession.ts:45-55`. Asymmetric diagnosability between player and host/display routes.

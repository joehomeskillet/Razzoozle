## 2. Reuse vs. parallel-implementation — condensed verdicts

Genuinely reused (single implementation, no dup): **AnswerButton** (`AnswerButton.tsx:62-116`, consumed by `ChoiceGrid` MP+Solo and `Responses.tsx`), **CircularTimer** (`CircularTimer.tsx:51-134`, MP+Solo+Question.tsx identical), **ListRow/Schülerzeile** (`console/ListRow.tsx`, 5 admin screens per W4/W6), **Card** as used by the join flow (`Card.tsx:9-20`, `join/Room.tsx`/`join/Username.tsx`), **RewardRow shell** (`RewardRow.tsx:37-38` `rewardCardClass`, consumed by both `ScoreToast.tsx` and the MP `RewardStack`), **Button primary/secondary variants** (`Button.tsx:31-38`, 64+ sites — with the SharePage exception noted above).

Genuinely parallel (separate hand-rolled implementations of the same visual concept), ranked by evidence strength:

1. **Medal (podium rank badge)** — `states/Podium.tsx:106-146` (uses `--tier-gold/silver/bronze` tokens, `size-20 md:size-26`) vs `features/results/SharePage.tsx:28-58` (hardcodes `bg-yellow-500`/`bg-gray-400`/`bg-amber-700`, `size-14 md:size-20`). Structurally identical (same sheen-overlay markup, same prop shape `{rank}`), independently written twice, one theme-able and one not.
2. **Dialog shell** — 5 independent Radix wrappers with different radius/overlay-opacity/close-button-placement: `components/AlertDialog.tsx:39` (`rounded-[var(--radius-theme)]`, `bg-black/40`), `components/manager/DialogPanel.tsx:36-42` (same tokens, different primitive), `features/game/components/GameWrapper/RejoinQrDialog.tsx:56-64` (`rounded-2xl`, `bg-black/70`), `states/Room.tsx:182-196` (`rounded-2xl bg-white`, no border/token), `states/Room.tsx:283-320` (kick-confirm — yet another variant, `bg-white text-black`, destructive button uses `--state-wrong` where `Button.tsx`'s own danger variant uses `--danger-bg`).
3. **Answer-screen orchestration** — `states/Answers.tsx` (724L) vs `states/SoloAnswers.tsx` (431L), self-documented dup (`SoloAnswers.tsx:1-9`). Already flagged in #19.
4. **ScoreBadge / result reveal** — toast (Solo, `ScoreToast.tsx`) vs full-screen state (`Result.tsx`, MP+Host+Player). Already flagged in #19 at the "5+ implementations" level; this audit adds the architectural framing.
5. **Leaderboard/result row** — `Leaderboard.tsx` / `SoloLeaderboard.tsx` / `TeamLeaderboard.tsx` / `SharePage.tsx` (4 files, no shared row component).
6. **StatusBadge** — 4 files, 2 token pairs (`--status-online-*` vs `--state-correct`).
7. **Switch** — `ToggleField.tsx` (canonical) vs 3 hand-rolled `role="switch"` markups with 3 different track sizes.
8. **Select** — `Select.tsx` (canonical, token-bound) vs 4 raw `<select>` with hardcoded gray/white literals.
9. **Primary Button** — `Button.tsx` (canonical) vs `SharePage.tsx`'s 2 hand-rolled primary buttons.
10. **Game-Code field** — 3 unrelated code-entry/display widgets in one feature (`PinInput`, `Room.tsx` static digits, `Room.tsx` satellite-pair input).

## 4. Klassenmodus-specific notes

Grepped `identifier` (the Klassenmodus tracking field) across the game/lobby
surfaces: it is captured on join (`join/Username.tsx:120-138`, free-text,
gated by server-sent `requireIdentifier`) but never displayed anywhere
downstream — not on the roster card (`Room.tsx`), not on the leaderboard row,
not on the result screen. No visual element in this audit changes shape or
content based on Klassenmodus; the mode only gates which fields are collected
at join time (see `docs/sdd/game-solo-multiplayer-refactor/05-class-mode-join-spec.md`
for the join-flow contract itself, out of this audit's visual scope).

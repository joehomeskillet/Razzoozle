# Reconnect-Garantie + Minimale Latenz — Plan

_Multi-agent research+analysis swarm, 2026-06-03._

## Summary

Razzia's clientId-keyed, full-state-snapshot reconnect (uuid in localStorage -> handshake.auth -> PLAYER/MANAGER.RECONNECT -> SUCCESS_RECONNECT replay) is the correct, industry-standard foundation and is strictly more robust than socket.io transport-level Connection State Recovery for a venue quiz (it survives page reloads, network switches, and longer outages that defeat CSR). The highest-value moves are NOT adopting a new protocol or CSR, but closing five concrete app-level gaps verified in the code: (1) answer dedup keyed by volatile socket.id instead of clientId (round-manager.ts:387,400) so a tap lost to a blip then re-sent after reconnect can be dropped or mis-keyed; (2) fire-and-forget answer emit with no ack/outbox (Answers.tsx:66,80) so a tap during a wifi blip is silently lost; (3) getReconnectInfo() returns only {current,total} (round-manager.ts:107) so a mid-question reconnect lands on a dead screen and cannot answer; (4) the disconnect/reconnect race in reconnectPlayer/reconnectManager (game/index.ts) rejects a fast legitimate reconnect with RESET because connected is still true; (5) no per-player rejoin TTL and all state is in-memory so a server restart wipes every game. Latency wins are cheap and free: disable perMessageDeflate on the tiny hot-path frames, add tcp_nodelay + proxy_buffering off in nginx, lower ping heartbeats, add reconnectionDelayMax jitter cap, and (optional) swap to msgpack parser. SKIP WebTransport/QUIC, WebRTC datachannel, MQTT/NATS/CoAP, CSR, sticky-sessions/Redis as over-engineering at tens-to-hundreds players. Note: the analysis claim that startTime is set 'after broadcast' is inaccurate — it is set immediately before the SELECT_ANSWER broadcast; the real fairness issue is timeToPoint() scoring on server-RECEIVE time, which charges each player their own RTT.

## Reconnect-Garantie (priorisiert)

### 1. Make answer submission idempotent and durable on a STABLE key. Change dedup/lookup in selectAnswer from socket.id to the player's clientId + currentQuestion index. selectAnswer currently does players.findById(socket.id) and dedups via playersAnswers.find(a => a.playerId === socket.id); after a reconnect updateSocketId() rewrites player.id, so a retried answer can target a stale/new socket and be dropped or double-evaluated. Resolve the player via clientId (socket.handshake.auth.clientId -> findByClientId), store playersAnswers entries keyed by {clientId, questionIndex}, and early-return on a matching (clientId, questionIndex) so retries are safe (effectively-once). This is the single change that GUARANTEES a tap survives a blip without losing the point.
- **Files:** /nvmetank1/projects/rahoot/source/packages/socket/src/services/game/round-manager.ts (selectAnswer ~387-421, playersAnswers shape); /nvmetank1/projects/rahoot/source/packages/socket/src/services/game/player-manager.ts (findByClientId already exists)
- **Effort:** M (half a day; touches Answer type + scoring loop which already maps playerAnswer by player.id — switch to clientId)
- **Why:** socket.id is volatile across reconnect (updateSocketId rewrites player.id); keying scoring/dedup on it is the root cause of an answer being lost or mis-attributed exactly in the wifi-blip case the goal targets. clientId is the durable identity already in the handshake.

### 2. Client-side at-least-once answer with ack + tiny outbox. Wrap the SELECTED_ANSWER emit in Answers.tsx as socket.timeout(2000).emit(EVENTS.PLAYER.SELECTED_ANSWER, payload, ackCb). Before emit, write the pending answer to sessionStorage keyed by gameId+questionNumber; clear it on ack. On the existing useEvent('connect') handler in party/$gameId.tsx, flush any un-acked pending answer for the current question (re-emit with same key — server is now idempotent from rank 1). Server: have selectAnswer invoke the ack callback so the client can clear the outbox and show a confirmed 'answer locked in' state instead of just a spinner.
- **Files:** /nvmetank1/projects/rahoot/source/packages/web/src/features/game/components/states/Answers.tsx (handleAnswer:61-71, submitSlider:73-86); /nvmetank1/projects/rahoot/source/packages/web/src/pages/party/$gameId.tsx (connect handler:30-33); /nvmetank1/projects/rahoot/source/packages/socket/src/services/game/round-manager.ts (selectAnswer ack); /nvmetank1/projects/rahoot/source/packages/socket/src/handlers/game.ts (SELECTED_ANSWER wiring to pass ack)
- **Effort:** M (outbox + ack plumbing, ~half day)
- **Why:** Today the emit is pure fire-and-forget; a blip during the answer window silently drops the tap and the player loses points/place. Ack+retry+outbox converts best-effort to effectively-once and gives the player a confirmed receipt.

### 3. Enrich the reconnect snapshot so a mid-question blip can still answer. getReconnectInfo() returns only {current,total}. While the round is in SELECT_ANSWER, the SUCCESS_RECONNECT payload must include the full answerable question (question text, type, answers OR slider min/max/step/unit, media) PLUS a server-computed remainingTime = startTime + question.time*1000 - Date.now() (clamped >=0), AND whether this clientId already has a recorded answer for the current question. The cached playerStatus snapshot already partially does this if the player saw SELECT_ANSWER before dropping — verify it survives onNewQuestion()'s playerStatus.clear() and that a player who joined-late/blipped before receiving SELECT_ANSWER still gets a renderable question.
- **Files:** /nvmetank1/projects/rahoot/source/packages/socket/src/services/game/round-manager.ts (getReconnectInfo:106-111, must expose current question + remaining time); /nvmetank1/projects/rahoot/source/packages/socket/src/services/game/index.ts (reconnectPlayer:184-227 builds the SUCCESS_RECONNECT status)
- **Effort:** M
- **Why:** A player who drops during the SELECT_ANSWER window reconnects into a 'question N of M' shell with no options and no timer, so they cannot submit and lose the round even though the socket recovered.

### 4. Fix the disconnect/reconnect race: prefer takeover over rejection. reconnectPlayer rejects with GAME.RESET 'playerAlreadyConnected' when player.connected is still true; reconnectManager does the same for the manager. A fast flaky-wifi reconnect that arrives BEFORE the server processed the old socket's 'disconnect' wrongly ejects a legitimate player/manager. When the incoming clientId matches an existing player/manager but the incoming socket.id differs, TAKE OVER: swap socket.id (updateSocketId), set connected=true, move the cached playerStatus to the new socket, and proceed with SUCCESS_RECONNECT instead of RESET.
- **Files:** /nvmetank1/projects/rahoot/source/packages/socket/src/services/game/index.ts (reconnectPlayer:184-227 'if (player.connected) ... RESET'; reconnectManager:155-182 'if (this._manager.connected) ... RESET')
- **Effort:** S (guard inversion + socket swap)
- **Why:** On flaky wifi the reconnect frequently races ahead of the disconnect event; the current guard turns a recoverable blip into a hard eviction with full state loss — directly defeats the guarantee.

### 5. Add an explicit, bounded per-player rejoin grace window and a manager-disconnect signal. Today setPlayerDisconnected only flips connected=false with no deadline (player slot lives until the whole game is cleaned via the 5-min EMPTY_GAME_TIMEOUT, which only triggers when the MANAGER is gone). Make the player-rejoin grace explicit (e.g. 2-3 min, configurable) and surface 'player reconnecting…' vs 'player left' to the manager UI. Separately, when the manager disconnects mid-game, broadcast MANAGER.DISCONNECTED to players (and MANAGER.RECONNECTED on return) so the game isn't a phantom; keep the existing 5-min manager grace but make it configurable.
- **Files:** /nvmetank1/projects/rahoot/source/packages/socket/src/services/registry.ts (EMPTY_GAME_TIMEOUT_MINUTES:24, cleanupEmptyGames:175); /nvmetank1/projects/rahoot/source/packages/socket/src/services/game/player-manager.ts (setDisconnected:88); /nvmetank1/projects/rahoot/source/packages/socket/src/services/game/index.ts (setManagerDisconnected:230); /nvmetank1/projects/rahoot/source/packages/socket/src/handlers/game.ts (handleManagerLeave:14, disconnect:151)
- **Effort:** S-M
- **Why:** Score/place must never be GC'd mid-game while a phone is briefly backgrounded; an explicit tunable grace window makes the guarantee observable and bounded, and players need to know the manager is merely reconnecting, not gone.

### 6. CONDITIONAL — survive a server restart mid-game. All games live in the in-memory Registry singleton and vanish on deploy/crash, so no clientId reconnect can recover them regardless of any other fix. If live deploys/crashes during a quiz are a real venue risk, persist a lightweight snapshot of Registry state (players, points, streak, currentQuestion, startTime, status) to disk on a short interval and on the existing SIGTERM/SIGINT handlers (index.ts already has them), and reload on boot. This is far cheaper than multi-node Redis and directly serves 'guarantee rejoin after ANY connection loss'.
- **Files:** /nvmetank1/projects/rahoot/source/packages/socket/src/services/registry.ts (cleanup/snapshot); /nvmetank1/projects/rahoot/source/packages/socket/src/index.ts (SIGTERM/SIGINT:43-50, flush on shutdown + reload on boot)
- **Effort:** M-L
- **Why:** In-memory-only state is the one failure mode no socket-layer or clientId fix can cover; only worth building if mid-game restarts actually happen — otherwise defer.

## Latenz-Minimierung

### 1. Disable WebSocket compression on the hot path. Change perMessageDeflate: { threshold: 1024 } to perMessageDeflate: false in the ServerIO options. Answer/status/leaderboard frames are sub-1KB; compressing them adds per-message CPU + buffering latency with no size win, and the threshold:1024 already means they're never compressed anyway — so this only removes the negotiation/handshake overhead and any borderline frames.
- Files: /nvmetank1/projects/rahoot/source/packages/socket/src/index.ts:13-18 · Effort: S (one line) · Gain: Small but free: removes deflate negotiation + eliminates any compression buffering on borderline frames; a few ms and lower CPU under load. Zero risk.

### 2. Eliminate Nagle/delayed-ACK coalescing through nginx. Add `tcp_nodelay on;` and `proxy_buffering off;` (and `proxy_set_header X-Forwarded-For`) to the /ws location. Node enables TCP_NODELAY on its own sockets by default, but the nginx upstream proxy can still coalesce small frames; proxy_buffering off flushes them immediately rather than batching.
- Files: /nvmetank1/projects/rahoot/source/docker/nginx.conf (location /ws block, lines ~28-36 — currently sets proxy_read_timeout/proxy_send_timeout 3600s but no tcp_nodelay or proxy_buffering off) · Effort: S (two lines) · Gain: Removes the classic ~40ms Nagle+delayed-ACK stall on small frames behind the proxy. Free, zero risk; the most impactful raw-latency line.

### 3. Make scoring RTT-fair (lag compensation). timeToPoint(startTime, question.time) computes points from Date.now() at SERVER-RECEIVE time, so a +250ms venue-wifi player is systematically penalized and a retried answer (rank-2 outbox) is scored at its later arrival. Add a one-round-trip clock-sync on connect (clock:ping{t0} -> clock:pong{serverNow} -> client computes offset+halfRTT). Client sends an answeredAt offset relative to its SELECT_ANSWER receipt with the answer; server scores on that, CLAMPED to [serverHalfRTT, question.time] so a client can never claim it answered faster than physically possible (trust-but-verify). Keep startTime server-authoritative.
- Files: /nvmetank1/projects/rahoot/source/packages/socket/src/utils/game.ts (timeToPoint:60-71); /nvmetank1/projects/rahoot/source/packages/socket/src/services/game/round-manager.ts (selectAnswer:387-405, startTime); /nvmetank1/projects/rahoot/source/packages/web/src/features/game/components/states/Answers.tsx (send answeredAt); /nvmetank1/projects/rahoot/source/packages/web/src/features/game/contexts/socket-context.tsx (clock-sync on connect) · Effort: M · Gain: Removes the entire last-mile-RTT penalty from scoring (commonly 100-400ms of unfair point decay on bad wifi) and fixes the retry-penalty. The biggest FAIRNESS win; do it only if scoring-on-bad-wifi is a real complaint, gated by the clamp guardrail.

### 4. Tune heartbeats + reconnection backoff. Server: pingInterval ~10000, pingTimeout ~8000 (down from default 25000/20000) so a genuinely dead phone is reaped in ~18s instead of ~45s, freeing the slot for a clean reconnect (nginx proxy_read_timeout is 3600s, comfortably above either window — verified, no change needed there). Client: add reconnectionDelayMax: 5000 to cap exponential backoff so 100 phones reconnecting after a shared-AP blip don't thundering-herd on the same 1s tick (randomizationFactor default already jitters).
- Files: /nvmetank1/projects/rahoot/source/packages/socket/src/index.ts (ServerIO options:13-18); /nvmetank1/projects/rahoot/source/packages/web/src/features/game/contexts/socket-context.tsx (io() options:97-105, add reconnectionDelayMax) · Effort: S · Gain: Faster dead-peer reaping (~45s -> ~18s) shrinks the rejoin gap; backoff cap bounds worst-case reconnect latency under a venue-wide blip. Cheap, low risk (slightly more battery/keepalive traffic).

### 5. Index player + answer lookups (O(n) -> O(1)). selectAnswer does players.findById (linear scan) and playersAnswers.find (linear scan) on the hot path; at 100+ concurrent answers this is O(n^2)-ish per round. Back PlayerManager with a Map keyed by socket.id (maintained in join/updateSocketId/remove) and store playersAnswers as a Map keyed by clientId (aligns with rank-1).
- Files: /nvmetank1/projects/rahoot/source/packages/socket/src/services/game/player-manager.ts (findById:110, join/updateSocketId/remove); /nvmetank1/projects/rahoot/source/packages/socket/src/services/game/round-manager.ts (playersAnswers:50, selectAnswer:387-405) · Effort: S-M · Gain: Negligible at tens of players; measurable only at the upper hundreds. Worth folding into rank-1's Map refactor for free; not worth doing standalone.

### 6. OPTIONAL — swap to socket.io-msgpack-parser. Add the parser dependency to both @razzia/socket and @razzia/web, set `parser` on the Server (index.ts) and the client io() call. Shrinks already-tiny JSON modestly and parses faster; must be configured on BOTH ends or connections break.
- Files: /nvmetank1/projects/rahoot/source/packages/socket/src/index.ts; /nvmetank1/projects/rahoot/source/packages/web/src/features/game/contexts/socket-context.tsx; both package.json files · Effort: S (but measure first) · Gain: Modest for tiny JSON payloads; a few bytes/ms per message. Clean low-risk polish, not a game-changer. Do last, measure before/after.

## Protokoll-Urteile

- **Socket.IO Connection State Recovery (CSR)** — SKIP (do not enable): Redundant with Razzia's clientId snapshot-replay and strictly weaker for this use case: CSR is documented to fail on wifi<->cellular and AP<->AP switches (socketio discussion #5248), does not survive a server restart, regenerates the session id, and silently fails for long-lived connections (#5282) — exactly the venue failure modes the goal targets. It only buffers events for a bounded window in memory, adding per-socket overhead. Razzia's app-level resync already survives page reloads, network switches, and longer outages. The highest-value socket.io move here is to NOT adopt its headline feature. (If a sub-second smoothing safety net is ever wanted it can be layered later, but it must never be the source of truth — the in-memory Registry + clientId is.)
- **WebTransport / HTTP-3 / QUIC** — SKIP: Its superpowers (multi-stream HOL-blocking fix, unreliable datagrams) are unusable for Razzia: the hot path is a single must-not-lose, must-not-reorder answer per round on one logical channel — reliable+ordered, which WebSocket already gives. Adopting it means terminating HTTP/3+QUIC (UDP/443) in front of Node, which the current nginx /ws WebSocket proxy does NOT do (new QUIC proxy, cert handling, UDP firewall rules). Safari (a large share of phone players) is only partial/experimental, so you'd silently fall back to WebSocket for many players and maintain that path anyway. QUIC connection-migration is the one attractive property for rejoin, but the clientId app-layer session already makes rejoin loss-free on any transport — making migration a marginal convenience, never a requirement.
- **WebRTC DataChannel** — SKIP outright: WebRTC is a peer-to-peer model (SCTP over DTLS/ICE) needing signaling, STUN/TURN, and NAT traversal. Razzia is strictly client-to-central-authoritative-server (star topology with server-side scoring). You'd only use the data channel as an awkward WebSocket substitute, importing large complexity (ICE/TURN) for zero benefit and breaking the authoritative-server scoring model.
- **MQTT / NATS / CoAP (IoT brokers)** — SKIP: These are constrained-device / massive-scale protocols; Razzia runs full browsers at tens-to-low-hundreds, ~1-2 orders of magnitude below a single socket.io node's unaided ceiling. Browsers can't speak raw MQTT/NATS — they tunnel over WebSocket anyway, so none removes the flaky-WebSocket failure mode that is the actual problem; each only adds a stateful broker to deploy, secure, and persist at the venue. CoAP is doubly disqualified (browsers can't open raw UDP sockets). The one good idea (broker-queued persistent sessions keyed to a client identity) Razzia already implements at the app layer via clientId + SUCCESS_RECONNECT replay; MQTT QoS would only protect individual in-flight messages, not score/place, which lives in server game state.
- **Binary / msgpack parser (socket.io-msgpack-parser)** — ADOPT optionally (low priority, measure first): Unlike the exotic transports, this is a clean drop-in: same dependency on both ends, set parser on Server and client io(). It shrinks already-tiny JSON modestly and parses faster, low-risk. But the absolute byte/ms savings on Razzia's small answer/score payloads are small, it adds a client-bundle dependency, and it MUST be configured on both ends or connections break. Treat as final polish after the free wins (perMessageDeflate off, tcp_nodelay), not a priority — and benchmark before/after to confirm it's worth the added dependency.

## Bewusst NICHT (over-engineering)

- Enabling socket.io connectionStateRecovery — redundant with the clientId snapshot-replay, weaker (fails on network switches / server restart / long-lived sockets), and adds per-socket memory overhead. The current architecture is correct; double down on it instead.
- WebTransport/QUIC, WebRTC datachannel, MQTT/NATS/CoAP — all solve problems Razzia doesn't have (HOL-blocking, unreliable datagrams, constrained devices, massive scale) while adding real infrastructure (QUIC proxy, STUN/TURN, a stateful broker) and, for browser clients, still riding over WebSocket anyway. Pure over-engineering at this scale.
- Sticky sessions + Redis adapter + multi-node scaling — a single Node socket.io process handles thousands of connections; tens-to-hundreds of players have ample headroom. The single in-memory Registry is an ASSET for the rejoin guarantee (no cross-node state to lose). Revisit only if a single quiz ever needs multiple WS nodes.
- Full offline-first / CRDT-style sync engine or a generic durable client action queue — the only action worth queuing is the single in-flight answer until the question deadline; a tiny sessionStorage outbox (rank-2) covers it. A general sync engine is large complexity for a time-boxed, server-authoritative quiz.
- Per-event sequence numbers + ordered event-replay protocol — the full-state STATUS snapshot on reconnect is simpler and correct for a turn-based quiz; you need 'what screen am I on right now', not ordered replay of every missed tick.
- Client-side prediction + server reconciliation, snapshot/entity interpolation, deterministic lockstep, GGPO rollback — continuous-state and P2P-determinism techniques that are a categorical mismatch for tapping one of four answers; zero payoff, large complexity.
- Pinning transports:['websocket'] only (dropping polling) — a public-venue audience can hit captive-portal/corporate wifi that blocks raw WebSocket; keep polling as the safety-net fallback. engine.io already tries websocket-first, so the gain from removing polling is marginal and the downside (a player who can't connect at all) is severe.
- Jitter buffer / 1-2s grace window after cooldown for late answers (from analysis) — at odds with the first-correct-bonus fairness and adds timing complexity; the rank-1 idempotency + rank-2 ack/outbox + rank-3 lag-compensated scoring already make a blipped answer survive and be scored fairly without holding the round open. Skip the extra buffer.
- Lowering the manager-empty cleanup below ~2 min or aborting the game on brief manager disconnect — too aggressive for venue wifi; a manager phone backgrounding for 60s would kill a live game. Keep a generous (configurable) manager grace and just SIGNAL disconnect to players (rank-5) rather than tearing down.
